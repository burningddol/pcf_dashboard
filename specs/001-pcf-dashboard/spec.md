# Spec: PCF Dashboard

**Feature**: 001-pcf-dashboard
**Timebox**: 2일(48시간) 집중 작업, 5일 여유
**평가 영역**: 도메인 이해(25%) · 시스템 설계(30%) · UX(20%) · 논리적 설명(20%)

---

## 1. 배경

기업이 자사의 **제품 탄소 발자국(PCF, Product Carbon Footprint)** 을 월 단위로
관리하려면 활동 데이터(전기·원소재·운송)를 입력하고, 배출계수를 곱한 결과를
GHG Scope별로 집계해 한눈에 볼 수 있어야 한다. 본 대시보드는 그 최소 단위를
구현한다.

대상자:
- **경영자** — 월별 추세와 Scope 비중을 빠르게 파악
- **실무자** — 활동 데이터를 직접 입력하고 즉시 PCF를 확인

두 페르소나가 같은 화면을 쓰되, 페르소나 토글 같은 분기를 두지 않고 단일
위계의 레이아웃으로 충분하도록 설계한다.

---

## 2. User Stories

### US1 (P1) — PCF 대시보드 Overview

**As a** 경영자 또는 실무자,
**I want** 회사·기간으로 필터링된 PCF 배출량을 한 화면에서 보고,
**So that** 회사의 탄소 배출 추세와 Scope 비중을 즉시 파악할 수 있다.

화면 구성:
- KPI 카드 3개 — 총 배출량(`tCO₂e`), Scope 2/Scope 3 비중, 전월 대비 변화율
- 월별 Stacked Bar — 활동 source(전기·플라스틱1·플라스틱2·운송)별 색상 구분
- 필터 바 — 회사 선택, 기간(YYYY-MM 시작·종료)
- Loading 스켈레톤 / Error 메시지 + 재시도 / Empty(필터 결과 없음) 안내

**Acceptance Scenarios**

1. **기본 진입**
   - Given 시드 데이터가 있는 상태에서 사용자가 `/`에 접속한다
   - When 화면이 로딩된다
   - Then 첫 렌더에 스켈레톤이 표시되고, 데이터 도착 후 KPI 카드 3개와 Stacked
     Bar가 표시된다. 모든 숫자에 `tCO₂e` 단위가 같이 표기된다.

2. **회사·기간 필터**
   - Given 사용자가 회사를 `Acme Corp`로 선택한다
   - When 기간을 `2025-03 ~ 2025-08`로 좁힌다
   - Then KPI와 차트가 해당 회사·기간 데이터만 반영하도록 즉시 갱신된다.

3. **빈 결과**
   - Given 사용자가 데이터가 없는 기간을 선택한다
   - When 필터를 적용한다
   - Then KPI는 `0 tCO₂e`로 표시되고, 차트 영역에는 "선택한 기간에 데이터가
     없습니다" 안내가 표시된다.

---

### US2 (P1) — 활동 데이터 입력 및 PCF 확인

**As a** 실무자,
**I want** 신규 활동 데이터(전기·원소재·운송)를 입력하면 즉시 tCO₂e가 계산되어
테이블에 반영되고,
**So that** 데이터 입력과 결과 검증을 한 화면에서 끝낼 수 있다.

화면 구성:
- 활동 데이터 테이블 — source · 기간(`YYYY-MM`) · 활동량 · 단위 · `tCO₂e` · Scope
- 신규 추가 폼 — 활동 유형, 기간, 활동량, 단위 입력
- PCF 프리뷰 — 입력 중 `활동량 × 배출계수`를 실시간 표시
- 행 클릭 시 계산 상세 패널 — 적용된 계산식과 배출계수 버전(`v1` 등) 표시
- 인라인 검증 — 음수, 빈 값, 필수 필드 누락 시 즉시 에러 메시지

**Acceptance Scenarios**

1. **정상 입력 (낙관적 업데이트)**
   - Given 사용자가 폼에 `전기 / 2025-09 / 12000 / kWh`를 입력한다
   - When 제출 버튼을 누른다
   - Then 테이블에 즉시 새 행이 추가되고, 200~800ms 지연 뒤 서버 응답이 오면
     성공 토스트가 표시된다.

2. **서버 실패 시 롤백**
   - Given 사용자가 정상 데이터를 입력한다
   - When 가짜 백엔드가 15% 확률로 실패를 반환한다
   - Then 낙관적으로 추가됐던 행이 자동으로 제거되고, "추가에 실패했어요.
     다시 시도해 주세요" 토스트가 표시된다.

3. **검증 에러**
   - Given 사용자가 활동량에 `-5`를 입력한다
   - When 폼이 포커스를 잃거나 제출을 시도한다
   - Then "활동량은 0 이상이어야 합니다" 인라인 메시지가 표시되고, 제출
     버튼은 비활성화된다.

4. **계산 상세 확인**
   - Given 테이블에 활동 데이터가 있다
   - When 사용자가 한 행을 클릭한다
   - Then 사이드 패널에 `12,000 kWh × 0.4781 kgCO₂e/kWh = 5,737.2 kgCO₂e
     (≈ 5.74 tCO₂e)` 형태의 계산식과 적용 배출계수 버전 `v1`이 표시된다.

## 3. Functional Requirements

| # | 요구사항 |
|---|---|
| FR-01 | 활동 데이터를 회사·기간(YYYY-MM)으로 필터링하여 조회한다 |
| FR-02 | 활동량과 배출계수로부터 tCO₂e를 계산해 응답에 포함한다 |
| FR-03 | 활동 유형을 GHG Scope(2 또는 3)에 매핑하여 표시한다 |
| FR-04 | 월별 합계와 source별 합계를 각각 집계해 KPI·차트에 사용한다 |
| FR-05 | 전월 대비 변화율을 계산하고, 비교 불가 시 "—"로 표시한다 |
| FR-06 | 모든 배출량 숫자에 `tCO₂e` 단위를 함께 표기한다 |
| FR-07 | 신규 활동 데이터 추가 시 낙관적 업데이트 후 실패 시 롤백한다 |
| FR-08 | 가짜 백엔드는 200~800ms 지연 + 쓰기 15% 실패를 시뮬레이션한다 |
| FR-09 | 입력 폼은 음수·빈 값·기간 형식 오류를 인라인으로 검증한다 |
| FR-10 | 행 클릭 시 계산식과 적용 배출계수 버전을 보여준다 |
| FR-11 | 데이터 의존 화면은 Loading/Error/Empty 3상태를 모두 처리한다 |
| FR-12 | 배출계수가 갱신되어도 과거 활동의 tCO₂e는 변하지 않는다 (factorId 고정) |

## 4. Success Criteria

| # | 기준 |
|---|---|
| SC-01 | 시드 데이터(2회사 × 8개월 × 4유형 = 64행) 로딩 후 첫 렌더가 1초 이내 |
| SC-02 | KPI·차트·테이블이 동일한 필터 결과에서 수치적으로 일치한다 |
| SC-03 | 신규 입력 후 5초 이내 토스트(성공 또는 롤백)가 사용자에게 도달한다 |
| SC-04 | 핵심 도메인 함수(`computeTCO2e`, `mapToScope`, `aggregateByMonth`,
        `aggregateBySource`) Vitest 테스트가 모두 통과한다 |

---

## 5. Out of Scope

이번 과제 범위에서 명시적으로 제외한다 (이유 함께 기록).

- **Excel 직접 임포트** — 보너스 항목. 시간 여유가 있을 때 추가 (T36 후보)
- **Sankey/Lifecycle 다이어그램** — Stacked Bar로 source 분해가 충분하다고 판단
- **페르소나 토글 UI** — 단일 위계의 화면으로 두 페르소나를 모두 충족 가능
- **감사추적(AuditEntry)** — 생성일(`createdAt`)만 보존, 변경 이력은 제외
- **이상치 감지 / 자동 추론** — 도메인 정확성을 위해 규칙 기반 매핑만 사용
- **인증·다중 사용자** — 채용과제 범위 밖
